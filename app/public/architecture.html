Executive Summary

Strategic Hybrid Architecture: AWS + Salesforce for Regulated Pharma

Takeda's GDDT C5 Patient App currently operates on a production-grade AWS data engineering backbone with a medallion lakehouse architecture (Kinesis → Glue → Bronze/Silver/Iceberg). This architecture proposal enriches the existing infrastructure with Salesforce's engagement intelligence layer — additive, not disruptive.

The hybrid approach positions AWS as the data engineering powerhouse (high-throughput ingestion, transformation, analytics) and Salesforce as the patient engagement orchestrator (360° profiles, AI-driven personalization, conversational AI, multi-channel communications). This guide details the architecture, security considerations, regulatory compliance, implementation strategy, and ROI analysis.

💪

Keep AWS Strengths

No rip-and-replace. The existing Kinesis → Glue → Iceberg pipeline continues operating exactly as before. AWS remains the data engineering backbone for high-throughput clinical data ingestion and analytics.

✨

Add SF Intelligence

Salesforce provides unified patient profiles (Data Cloud), real-time personalization, AI patient assistants (Agentforce), and consent-aware multi-channel communications — all gaps in the current AWS-only stack.

🔒

Pharma-Grade Security

Pseudonymization-by-design, Shield Platform Encryption, Field Audit Trail (21 CFR Part 11), OOTB consent management (GDPR/HIPAA), and Lake Formation governance — all regulatory requirements addressed.

🎯 Key Architectural Decision: Hybrid Approach

Why not all-AWS or all-Salesforce? This hybrid architecture preserves AWS strengths (high-throughput ETL, medallion lakehouse, Databricks-scale analytics) while adding Salesforce strengths (patient 360, AI personalization, conversational AI, multi-channel orchestration).

The S3/Iceberg Connector is the critical bridge — it enables zero-copy federation from AWS Lakehouse to Salesforce Data Cloud, meaning pseudonymous data flows seamlessly without duplication or streaming latency.

☁️ AWS Data Engineering Backbone: Detailed Analysis

1️⃣ Ingestion Layer: API Gateway → Lambda → Kinesis

API Gateway (REST)
  • • Endpoint: POST /v1/clinical-events
  • • Throttling: 10,000 RPS per region (burst: 5,000)
  • • Validation: JSON Schema validation at gateway (fail-fast)
  • • Authentication: JWT tokens (short-lived, app-specific)
  • • Regional: US, EU, APAC endpoints (GDPR compliance)

POST /v1/clinical-events

{
  "event_type": "medication_taken",
  "patient_uuid": "uuid-123...",
  "timestamp": "2026-07-09T14:30:00Z",
  "payload": {
    "medication_id": "MED-001",
    "dose_taken": true,
    "side_effects": []
  }
}
Lambda Enrichment
  • • Runtime: Python 3.12 (512 MB memory, 30s timeout)
  • • Enrichment: Adds region, care_program_id, device_type
  • • PII Vault: DynamoDB lookup for patient_uuid → PII mapping
  • • Business Rules: Age-based dosage validation, time-zone conversion
  • • DLQ: Failed events → SQS dead-letter queue

Enriched Event:

{
  ...original_fields,
  "enriched_metadata": {
    "region": "US-EAST-1",
    "care_program_id": "CP-TAKEDA-001",

    "enriched_at": "2026-07-09T14:30:01Z"

}
Kinesis Data Streams
  • • Sharding: 50 shards per stream (50,000 records/sec ingestion capacity)
  • • Retention: 7 days (compliance + replayability)
  • • Deduplication: Exactly-once delivery via patient_uuid + event_timestamp composite key
  • • Partitioning: By patient_uuid (ensures ordered processing per patient)
  • • Encryption: AES-256 at rest (AWS KMS), TLS 1.3 in transit

2️⃣ Transformation Layer: AWS Glue ETL

Glue Job Configuration
  • • Engine: PySpark 3.5 (DPU: G.2X workers, autoscaling 10-50)
  • • Trigger: Event-driven (Kinesis → S3 → Lambda → Glue)
  • • Schema Registry: AWS Glue Schema Registry (Avro schemas)
  • • Data Quality: Glue Data Quality rules (completeness, uniqueness, range checks)
  • • Checkpointing: S3-backed checkpoints (exactly-once semantics)
Transformation Logic
  • • Cleansing: Null handling, type coercion, outlier detection
  • • Deduplication: Window function over (patient_uuid, event_type, timestamp)
  • • Pseudonymization: Hash patient_uuid with SHA-256 + salt (Bronze → Silver boundary)
  • • Validation: FHIR R4 compliance checks for clinical data
  • • Partitioning: By date (year/month/day) + care_program_id

Glue PySpark Pseudonymization Snippet:

from pyspark.sql import functions as F
from hashlib import sha256

# Pseudonymize patient_uuid at Bronze → Silver boundary
def pseudonymize_uuid(uuid_val):
    salt = "TAKEDA-GDDT-SALT-2026"  # Stored in AWS Secrets Manager
    return sha256(f"{uuid_val}{salt}".encode()).hexdigest()

df_silver = df_bronze.withColumn(
    "patient_pseudo_id",
    F.udf(pseudonymize_uuid, StringType())(F.col("patient_uuid"))
).drop("patient_uuid")  # Remove PII

# Write to Silver with Iceberg MERGE (upsert)
df_silver.writeTo("silver.clinical_events") \
    .using("iceberg") \
    .tableProperty("write.format.default", "parquet") \
    .partitionBy("event_date", "care_program_id") \
    .createOrReplace()

3️⃣ Medallion Lakehouse: Bronze → Silver → Iceberg (Gold)

Bronze (Raw)
  • • Purpose: Immutable audit trail
  • • Data: Raw JSON + PII (encrypted)
  • • Format: JSON (compressed with Snappy)
  • • Retention: 7 years (GxP compliance)
  • • Access: Restricted (IAM role: audit-only)
  • • Encryption: SSE-KMS (CMK: bronze-pii-key)
Silver (Clean)
  • • Purpose: Pseudonymous analytics-ready
  • • Data: Cleansed, deduplicated, NO PII
  • • Format: Parquet (columnar)
  • • Retention: 3 years (operational analytics)
  • • Access: Databricks, Athena (row-level filters)
  • • Quality: 99.9% completeness SLA
Iceberg (Gold)
  • • Purpose: Production-grade data product
  • • Data: Aggregated, feature-engineered
  • • Format: Apache Iceberg (ACID transactions)
  • • Features: Time-travel, schema evolution
  • • Access: Data Cloud (S3/Iceberg Connector)
  • • SLA: 15-min freshness

1️⃣ Integration Bridge: S3/Iceberg Connector

Zero-Copy Federation
  • • No Data Movement: Data Cloud reads directly from S3 (no ETL latency)
  • • Metadata Sync: Iceberg catalog → Data Cloud schema (every 15 min)
  • • Incremental Refresh: Only new/changed Iceberg snapshots synced
  • • Cost Efficiency: Pay only for S3 GET requests (no data duplication)
Configuration
  • • S3 Bucket: s3://takeda-gddt-iceberg-gold-us/
  • • IAM Role: Cross-account role (Salesforce → AWS)
  • • Tables Synced: patient_engagement_summary, medication_adherence, assessments
  • • Refresh Schedule: Every 15 minutes (near-real-time)

2️⃣ Data Cloud: Unified Individual (Patient 360)

Data Model Objects (DMOs)
  • • Unified Individual: patient_pseudo_id (from Iceberg) ↔ Person Account (Health Cloud)
  • • Custom DMOs: Assessment_Result__dlm, Wearable_Sample__dlm, Medication_Adherence__dlm
  • • Identity Resolution: Pseudo-ID matching via Data Graph (no PII in Data Cloud)
  • • Relationships: 1:N (Individual → Assessments, Individual → Medications)
Calculated Insights
  • • engagement_score__c: 0-100 score (7-day activity + adherence + assessment)
  • • adherence_risk__c: HIGH/MEDIUM/LOW (missed doses in last 14 days)
  • • assessment_overdue__c: Boolean (next assessment > 3 days overdue)
  • • Refresh: Hourly batch (Data Cloud Insights Engine)

Calculated Insight SQL (engagement_score__c):

SELECT
  patient_pseudo_id,
  (
    (adherence_pct * 0.4) +                    -- 40% weight
    (assessment_completion_rate * 0.3) +       -- 30% weight
    (app_sessions_last_7d / 7 * 10 * 0.3)      -- 30% weight (normalized)
  ) AS engagement_score__c
FROM iceberg_gold.patient_engagement_summary
WHERE event_date >= CURRENT_DATE - 7;

Pseudonymization Architecture Explained

🔴 HIGH SENSITIVITY

PII Isolation Zone

  • ► PII Vault (DynamoDB): Stores patient_uuid → {FirstName, LastName, Email, DOB, Phone}
  • ► KMS Encryption: Envelope encryption per-field with AWS KMS customer-managed keys
  • ► IAM Restriction: Only enrichment Lambda + re-identification service have read access
  • ► Bronze Layer: Raw messages with PII — 7-year retention for GxP audit trail
🟢 PSEUDONYMOUS ONLY

Pseudonymous Data Zone

  • ► Pseudonymization Step: Glue ETL replaces PII with patient_uuid before Silver layer
  • ► Silver + Iceberg: Contain ONLY patient_uuid — no names, emails, DOB
  • ► Data Cloud DMOs: Ingest pseudonymous data via S3/Iceberg Connector
  • ► Databricks Analytics: Queries Iceberg with pseudonymous keys only

🎯 Key Architectural Insight:

Layer 1 (Metadata Templates) is fully CI/CD friendly — you can deploy DataCalcInsightTemplate and DataKitObjectTemplate via sfdx force:source:deploy. How Personalization Works (Step-by-Step)

1

Data Graphs Provide Context

Profile Graph: Patient attributes from Unified Individual (diagnosis, treatment_stage, assessment_scores, program_enrollments, medication_adherence_rate). Catalog Graph: Content items (educational materials, program info, care guides) with metadata (topic, difficulty_level, content_type).

// Example Profile Graph Structure: { "patient_uuid": "abc123", "diagnosis": "Type 2 Diabetes", "treatment_stage": "Advanced Management", "assessment_score": 8.2, "last_assessment_date": "2026-07-01", "medication_adherence_rate": 0.85 } // Example Catalog Graph Structure: { "content_id": "edu-material-456", "title": "Managing Your Medication", "topic": "medication_adherence", "difficulty_level": "beginner", "content_type": "video" }
2

Calculated Insights Score Patient Behavior

Real-time SQL scoring runs on Data Cloud DMOs. Each Calculated Insight is a SQL query that computes a derived metric.

engagement_score__c
DECIMAL (0-1): Measures patient's app usage frequency + content interaction rate
-- SQL Logic: SELECT patient_uuid, (logins_last_30d / 30.0) * 0.5 + (content_views_last_30d / 10.0) * 0.5 AS engagement_score__c FROM Unified_Individual__dlm
adherence_risk__c
TEXT (low|med|high): Risk level based on medication adherence rate
-- SQL Logic: SELECT patient_uuid, CASE WHEN adherence_rate >= 0.8 THEN 'low' WHEN adherence_rate >= 0.5 THEN 'med' ELSE 'high' END AS adherence_risk__c FROM Medication_Adherence__dlm
assessment_overdue__c
BOOLEAN: TRUE if last assessment was > 30 days ago
-- SQL Logic: SELECT patient_uuid, CASE WHEN DATEDIFF(DAY, last_assessment_date, CURRENT_DATE()) > 30 THEN TRUE ELSE FALSE END AS assessment_overdue__c FROM Assessment_History__dlm
3

Recommender Engine Scores & Ranks Content

The Recommender combines Data Graphs + Calculated Insights to score each content item for relevance to the patient. Returns top N items ranked by score (0-1).

// Example Recommender Logic (pseudo-code): FOR EACH content_item IN Catalog_Graph: score = 0 // Match topic to patient diagnosis IF content_item.topic MATCHES patient.diagnosis: score += 0.4 // Boost if patient has high engagement IF patient.engagement_score__c > 0.7: score += 0.2 // Penalize if adherence risk is high (suggest support content) IF patient.adherence_risk__c == 'high' AND content_item.topic == 'medication_support': score += 0.3 // Boost if assessment overdue and content is about assessments IF patient.assessment_overdue__c == TRUE AND content_item.topic == 'assessments': score += 0.1 RETURN content_item WITH score

The Recommender returns a ranked list: [{contentId: "edu-456", score: 0.92}, {contentId: "edu-789", score: 0.78}, ...]

4

Personalization Points API Integration

Patient app calls the Personalization Points API to get real-time recommendations. The API endpoint is defined by the Point (Layer 3).

API Request
GET /services/data/v62.0/ personalizationPoints/ content_recommendations/decisions Headers: Authorization: Bearer {access_token} Query Params: patientId: {patient_uuid} maxResults: 5
API Response
{ "decisions": [ { "contentId": "edu-material-456", "score": 0.92, "title": "Managing Your Medication", "reason": "High adherence risk detected", "contentType": "video" }, { "contentId": "edu-material-789", "score": 0.78, "title": "Weekly Assessment Guide", "reason": "Assessment overdue (35 days)", "contentType": "article" } ] }

✅ Unified Consent Management Architecture

AuthorizationForm Object (Health Cloud)

All consent preferences are stored in the AuthorizationForm standard object in Health Cloud, ensuring a single source of truth.

  • • email_consent__c: Boolean (TRUE = can send email)
  • • sms_consent__c: Boolean (TRUE = can send SMS)
  • • rcs_consent__c: Boolean (TRUE = can send RCS)
  • • whatsapp_consent__c: Boolean (TRUE = can send WhatsApp)
  • • push_consent__c: Boolean (TRUE = can send push)
  • • inapp_consent__c: Boolean (TRUE = can show in-app)
  • • consent_date__c: DateTime (when consent was granted)
  • • consent_version__c: String (e.g., "v2.1" for audit trail)

Real-Time Consent Checks in Flows

Every Marketing Cloud Advanced Flow includes a Get Records step to fetch the patient's AuthorizationForm before sending.

Example Flow Logic (Pseudo-code):

TRIGGER: Data Cloud Segment "Adherence Risk = HIGH"

FLOW:
  1. Get Segment Members → [patient_pseudo_id list]

  2. FOR EACH patient_pseudo_id:
       a. Get Records: AuthorizationForm
          WHERE Patient__c = {patient_pseudo_id}

       b. Decision: sms_consent__c = TRUE?
          → YES: Send SMS via Marketing Cloud Advanced
                 Message: "Hi {{FirstName}}, reminder to log dose"
          → NO:  Log event (consent_denied) to Data Cloud
                 Skip send

  3. Wait 2 hours

  4. Decision: Check if dose logged (Data Cloud query)
     → NO: Escalate to Push Notification (if push_consent__c = TRUE)
     → YES: End flow (success)

⚠️ Regulatory Compliance: Consent Audit Trail

Every consent event is logged with Field History Tracking on AuthorizationForm, creating an immutable audit trail:
• WHO changed consent (patient via app, HCP via console, system automation)
• WHAT changed (email_consent__c: FALSE → TRUE)
• WHEN (timestamp with timezone)
This ensures GDPR Art. 7(1) (consent must be demonstrable) and 21 CFR Part 11 (audit trail) compliance.
